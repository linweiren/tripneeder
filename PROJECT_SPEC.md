# 0. 專案目標

建立一個 **臨時出門決策 AI Web App**

核心用途：

使用者臨時想出門
→ 不知道去哪
→ 快速生成 3 個可執行行程

支援：

* 半天到一天 (2~24 小時)
* 約會 / 放鬆 / 探索 等情境
* 預算控制
* 交通資訊
* 雨天備案
* 收藏方案

---

# 1. 產品邊界（嚴格限制）

本版本 **要做**

* 單日行程（2–24 小時）
* 三方案生成
* 預算估算
* 交通資訊

## Phase 7

登入註冊、點數制與 API key 移出前端

目的：

* 不是為了新增「儲存未來行程」功能
* 主要是為了管理 AI 使用量與點數
* 避免組員試用時無限制呼叫 API，造成成本失控
* 將 OpenAI API key 從前端 bundle 移出，避免正式試用時暴露金鑰

範圍：

* 登入 / 註冊
* 使用者身分識別
* 點數餘額
* 每次 AI 生成成功後扣點
* AI 失敗、JSON 格式錯誤、無法產生可解析結果時不扣點
* 建立 thin proxy / backend 呼叫 OpenAI
* OpenAI API key 只能存在 server-side secret / env，不可留在前端 bundle
* 前端 AI service 改為呼叫 proxy，不再直接呼叫 OpenAI Responses API
* proxy 需負責點數檢查、成功扣點與失敗不扣點的流程

# 溝通確認紀錄

以下內容為後續溝通確認後補充，若與前文有衝突，以本區最新確認內容為準。

## 2026-04-15 總覽補充

### AI 與模型

本專案會串接 OpenAI API，並使用 AI 做真實行程分析，不是單純 mock data 或純規則生成。

OpenAI 模型版本目前預設使用：

```ts
gpt-4.1-mini
```

若要更換模型，需先確認成本、結構化輸出穩定性與行程品質，不可未討論就自行更換。

2026-04-16 後續調整：MVP AI provider 改為 OpenAI API。前端呼叫 OpenAI Responses API，目前實測與開發以 OpenAI 為主，不再保留其他 AI provider 的實作。

OpenAI 設定：

```txt
VITE_OPENAI_BROWSER_CREDENTIAL="在此處填上你的openai api key"
VITE_OPENAI_MODEL="gpt-4.1-mini"
```

正式分享或發布前，OpenAI API key 不可留在前端，必須改走 thin proxy / backend。

OpenAI API key 設定先預留空值，填寫提示文字使用：

```txt
在此處填上你的openai api key
```

AI 呼叫必須包含明確 prompt，並嚴格定義回傳格式，避免前端解析失敗。

### 地點與範圍

支援範圍為全台灣，不限制城市或區域。地點分析主要交由 AI 根據使用者輸入判斷。

起點選擇只保留兩種：

1. 使用目前所在地，透過 browser geolocation 取得座標。
2. 使用自然語言輸入想去的地區或地點。

刪除地圖選點功能，不列入 MVP。

地點資料結構改為：

```ts
type TripLocation = {
  name: string
  lat?: number
  lng?: number
}
```

MVP Phase 1-5 主要使用 `name`，`lat` / `lng` 先預留，後續若要加深地圖能力再正式使用。

### Google Maps

MVP 只產生 Google Maps 外部連結，不串接 Google Maps API。Google Maps API、路線最佳化、地圖互動屬於雛型完成後的延伸功能。

### 交通方式

方案分析完成後，使用者可以選擇交通方式：

* 機車
* 開車
* 大眾運輸

切換交通方式會影響預估抵達時間。如果交通方式造成整體行程嚴重超時，介面需跳出警告，提醒此交通方式會導致行程時間過長。

交通時間估算規則：

* 機車：預設 40 km/hr
* 汽車：預設 60 km/hr
* 大眾運輸：使用網路或 AI 預估時間

### 預算與人數

預算輸入維持分類：

* 小資
* 一般
* 輕奢
* 豪華

方案結果顯示 AI 預估金額。

人數改為 1-10 人。若人數大於或等於 5，AI 分析時應避開精緻小巧、座位有限或不適合團體的店家與活動。

### 限制條件

`雨天` 不作為 tag，因為每個方案都應該有主方案與雨天備案。

`新鮮感` 不作為 MVP tag。

* 收藏方案

兩者皆以 localStorage 儲存。

### UI 與語言

介面語言使用全繁體中文。

首頁直接作為行程偏好輸入頁，不做行銷式 landing page。

UI 方向：

* 質感文青
* 字體優美
* 色調低飽和
* 簡單、直覺、快速
* 優先讓使用者一眼比較三方案

Figma MCP 可在設計階段參與，但不得影響核心功能完善度。若實作階段無法密集搭配 Figma MCP，先完成可用介面，後續再用 Figma MCP 重新整理 UI。

所有會呼叫 AI、可能等待較久的操作，都需要 loading 動畫或文字狀態，避免使用者以為畫面卡住。

### Stop 數量規則

AI 不可完全自由決定 stop 數量，必須遵守基礎規則：

* 2-4 小時：2-3 個 stop
* 4-8 小時：3-4 個 stop
* 8-12 小時：4-5 個 stop
* 12-24 小時：5-6 個 stop

依條件微調：

* 不要太累：-1
* 拍照優先：-1
* 美食優先：+1
* 5 人以上：-1

最後 stop 數量必須限制在 2 到 6 之間。

每個方案的 stop 需要有類型，不可只決定數量。至少包含：

* 主活動點
* 餐飲點
* 收尾 / 過渡點

避免產生同質性過高的行程。

### 發布與 PWA

最終目標是可在手機上執行的網頁式 App，並預計使用 PWA 方式讓同組組員能方便使用。

若發布流程不複雜可後續實作；若需要架構預留，開發時應先保留必要插口。

## 2026-04-15 總覽補充（二）

### OpenAI API 與前後端邊界

正式 API key 不可直接放在前端程式碼中。

本專案整體仍以前端 MVP 為主，但需要預留未來可切換為 thin proxy / backend 的結構。

AI 呼叫邏輯必須抽成獨立 service / interface，不可寫死在 React 元件內。

若開發初期為了速度暫時由前端直連 OpenAI API，可以接受，但程式結構必須能很容易切換成 proxy 模式。

正式分享或發布前，OpenAI API 必須改走 backend / proxy。

### AI 回傳格式

AI 回傳格式採用嚴格 JSON 解析。

OpenAI prompt 必須要求模型只回傳 JSON，不回傳 markdown、不回傳解釋文字、不包 code block。

前端需對 AI 回傳結果做解析與基本格式檢查。若解析失敗或格式不符，需顯示錯誤狀態，讓使用者可以重新分析。

### 大眾運輸估算

MVP 階段不串接 Google Maps API 或交通 API。

大眾運輸時間先由 OpenAI 根據常識與可推估資訊產生，UI 上應視為 AI 預估。

後續若導入 Google Maps API，再將交通時間改為真實資料。

### 交通超時警告

切換交通方式後，若造成整體行程時間過長，需警告使用者。

警告門檻：

* 總時間超過使用者設定時間的 20%
* 或總時間超過使用者設定時間 30 分鐘以上

兩者任一成立即顯示警告。

### PWA 階段

PWA 先列入 Phase 6 優化階段。

Phase 1-5 優先完成核心流程。Phase 6 再處理 mobile polish、manifest、icon、install prompt 等基礎 PWA 設定。

發布流程可後續處理，但專案結構需保持部署友善。

### Figma MCP 時機

早期開發先以功能成型為主。

核心流程穩定後，再使用 Figma MCP 重新整理視覺設計。

若 UI 顯示資訊、頁面布局、卡片資訊層級等規格不明確，每個 phase 開始前都必須先與使用者確認，不可自行大幅假設後直接實作。

## 2026-04-15 Phase 1 驗收修正

### 導覽與頁面入口

三種方案結果頁不可出現在主要導覽中，因為它必須是使用者完成偏好輸入並送出 AI 分析後才進入的畫面。

主要導覽改為預設關閉的三條線開合式選單，選單項目為：

* 行程規劃
* 收藏
* 最近生成

收藏與最近生成紀錄必須分成兩個頁面，不可放在同一個收藏頁中。

```txt

```

### Phase 2 輸入頁預先確認

輸入頁使用一頁式表單。

欄位預設值使用反灰 placeholder 文字，引導使用者輸入；使用者點擊準備輸入時，placeholder 會消失。

無法取得定位時，提示文字使用：

```txt
無法取得您的定位！請檢查權限設定。
```

時間輸入採更口語的呈現方式，例如「今晚 18:00 到凌晨 02:00」，而不是只呈現兩個生硬的 time input。

每個 phase 完成後，必須先讓使用者驗收。未經驗收通過，不可直接進入下一個 phase 的細節詢問或實作。

### Phase 1 驗收結果

## 2026-04-15 Phase 2 開工前確認

### 輸入頁形式

Phase 2 實作首頁一頁式表單，不做分步驟問答。

Phase 2 先完成表單資料收集、基本驗證、定位互動與 loading 骨架，不實際呼叫 OpenAI API，不產生三方案。

### 行程類型

行程類型使用可點選 chip / pill 按鈕，一次選一個。

選項包含：

* 約會
* 放鬆
* 探索
* 美食
* 戶外走走
* 室內活動
* 一個人

## 2026-04-16 Phase 5 驗收中 AI 生成修正：時間覆蓋與正餐落點

使用者確認針對 `gpt-4o-mini` 生成結果出現「使用者給定時間範圍與實際行程長度落差過大」與「餐廳過早排入、午餐語意不清」問題，採用以下修正方向：

## 2026-04-16 Phase 5 驗收中 AI 生成修正（二）：模型與點數制策略

使用者確認不採用固定時間 slot 骨架，因為每個景點合理停留時間與交通時間都會依地點性質不同而變動，不應由前端硬切固定時段。

最新修正方向：

* MVP 預設模型改為 `gpt-5-mini`，用於提升時間推理、規則遵守與行程品質。
* `.env` 與 `.env.example` 的 `VITE_OPENAI_MODEL` 改為 `gpt-5-mini`。
* 程式 fallback 預設模型也改為 `gpt-5-mini`。
* `vite.config.ts` 需同步注入 `VITE_OPENAI_MODEL`，避免本機 MVP 讀取模型設定不一致。
* 不採用背景自動重試，避免未來點數制下使用者一次操作被隱性消耗多次 API 成本。
* AI 仍可依景點性質自行分配 stop 停留時間與交通時間，不使用前端固定 slot。
* 時間約束採「彈性時間預算」策略，而不是固定時段骨架。
* 時間覆蓋率依行程長度分段：
  * 2-4 小時：至少使用 70%。
  * 4-8 小時：至少使用 75%。
  * 8-12 小時：至少使用 80%。
  * 12-24 小時：至少使用 70%。
* 提早結束容許值依行程長度分段：
  * 2-4 小時：最多提早 30 分鐘。
  * 4-8 小時：最多提早 45 分鐘。
  * 8-12 小時：最多提早 90 分鐘。
  * 12-24 小時：最多提早 180 分鐘。
* 驗證失敗時不顯示內部規則清單，改顯示產品文案，例如「這次產生的時間安排不夠完整，請重新分析一次。本次結果不會儲存。」
* 未來點數制建議只在成功產生可用結果時扣點；AI 回傳格式錯誤或時間驗證失敗時不扣點。

## 2026-04-16 Phase 5 驗收中 AI 生成修正（三）：避免時間診斷阻擋結果

使用者實測 `gpt-5-mini` 後仍出現「分析不完整」，確認主要問題不是單純模型能力，而是前端時間驗證過於一刀切：

* 目前一次產生 3 個方案，每個方案又包含一般行程與雨天備案，共 6 條時間線。
* 只要任一條時間線未達時間覆蓋率、餐期落點或 `totalTime` 診斷門檻，就會讓整批結果失敗。
* 三個方案時間不完全相同是合理結果，因保守型、平衡型、探索型本來就可能有不同節奏。
* MVP 不應把「時間診斷」視為硬性失敗，否則使用者會看不到 AI 已產生的可用內容。

最新修正方向：

* 模型預設改為 `gpt-4.1-mini`，作為成本與結構化輸出穩定性的折衷。
* `gpt-5-mini` 品質可能較好，但目前成本偏高，不作為 MVP 預設。
* `gpt-5-nano` 成本較低，但較不適合作為需要時間推理與行程品質的主要模型。
* 前端保留 JSON 結構、必要欄位、交通段關聯等硬性格式驗證。

### 點數制方向

點數制與 OpenAI API key 移出前端需排入登入 / 註冊後的核心開發範圍。

初步原則：

* 每位使用者需有點數餘額。
* 產生 AI 行程前需檢查點數是否足夠。
* 成功產生可解析結果後才扣點。
* OpenAI API 失敗、JSON 解析失敗、必要格式驗證失敗時不扣點。
* 不採用使用者看不到的背景自動重試，避免隱性消耗成本。
* 詳細扣點數量、免費點數、管理者補點方式、點數不足 UI 文案與流程需在該階段開工前另行確認。

### OpenAI API key 移出前端

目前 MVP 為了開發速度，仍使用前端直連 OpenAI Responses API，並由 `vite.config.ts` 在本機注入 `VITE_OPENAI_BROWSER_CREDENTIAL`。此作法只允許用於本機 MVP 開發，不可用於正式分享或組員試用。

最新確認後，OpenAI API key 移出前端需正式排入 Phase 7，而不是只作為發布前提醒。

Phase 7 需處理：

* 建立 thin proxy / backend 入口，讓前端不再直接呼叫 OpenAI。
* OpenAI API key 僅能存在 server-side secret / env。
* 前端 bundle 不可包含 OpenAI API key 或可被使用者讀取的 OpenAI credential。
* 前端既有 AI service 需保留抽象層，改為呼叫 proxy。
* proxy 需與點數制整合，負責產生前檢查點數、成功後扣點、失敗時不扣點。
* 正式發布與組員試用前，必須完成此項，否則不可發布。

* 原因是收藏時保存的是包含目前排序 / 交通切換狀態的 snapshot plan，但收藏狀態檢查原本拿原始 selected plan 比對。
* 兩者 fingerprint 不一定相同，因此成功收藏後可能又被重算為未收藏。

本機修正：

* 詳情頁收藏成功後不再顯示右側狀態文字。
* 收藏狀態檢查改用實際會保存的 snapshot plan。
* 收藏成功後按鈕維持 `已收藏` 並 disabled。
* 收藏失敗時才用 App dialog 顯示錯誤，並恢復按鈕可點狀態。

本機驗證：

* `npm run lint` 通過。
* `npm run build` 通過。
