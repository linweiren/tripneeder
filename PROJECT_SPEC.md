# 0. 專案目標

建立一個 **臨時出門決策 AI Web App**

核心用途：

使用者臨時想出門
→ 不知道去哪
→ 快速生成 3 個可執行行程

支援：

* 半天到一天 (2~24 小時)
* 約會 / 放鬆 / 探索 等情境
* 預算控制
* 交通資訊
* 雨天備案
* 收藏方案

---

# 溝通確認紀錄

以下內容為後續溝通確認後補充，若與前文有衝突，以本區最新確認內容為準。

## 2026-04-15 總覽補充

### AI 與模型

本專案會串接 Gemini API，並使用 AI 做真實行程分析，不是單純 mock data 或純規則生成。

Gemini 模型版本固定使用：

```ts
gemini-2.5-flash
```

不可使用 Gemini 1.5、2.0 或其他舊模型版本。

API key 設定先預留空值，填寫提示文字使用：

```txt
在此處填上你的api key
```

AI 呼叫必須包含明確 prompt，並嚴格定義回傳格式，避免前端解析失敗。

### 地點與範圍

支援範圍為全台灣，不限制城市或區域。地點分析主要交由 AI 根據使用者輸入判斷。

起點選擇只保留兩種：

1. 使用目前所在地，透過 browser geolocation 取得座標。
2. 使用自然語言輸入想去的地區或地點。

刪除地圖選點功能，不列入 MVP。

地點資料結構改為：

```ts
type TripLocation = {
  name: string
  lat?: number
  lng?: number
}
```

MVP Phase 1-5 主要使用 `name`，`lat` / `lng` 先預留，後續若要加深地圖能力再正式使用。

### Google Maps

MVP 只產生 Google Maps 外部連結，不串接 Google Maps API。Google Maps API、路線最佳化、地圖互動屬於雛型完成後的延伸功能。

### 交通方式

方案分析完成後，使用者可以選擇交通方式：

* 機車
* 開車
* 大眾運輸

切換交通方式會影響預估抵達時間。如果交通方式造成整體行程嚴重超時，介面需跳出警告，提醒此交通方式會導致行程時間過長。

交通時間估算規則：

* 機車：預設 40 km/hr
* 汽車：預設 60 km/hr
* 大眾運輸：使用網路或 AI 預估時間

### 預算與人數

預算輸入維持分類：

* 小資
* 一般
* 輕奢
* 豪華

方案結果顯示 AI 預估金額。

人數改為 1-10 人。若人數大於或等於 5，AI 分析時應避開精緻小巧、座位有限或不適合團體的店家與活動。

### 限制條件

`雨天` 不作為 tag，因為每個方案都應該有主方案與雨天備案。

`新鮮感` 不作為 MVP tag。

### 收藏與最近

收藏頁包含：

* 收藏方案
* 最近生成方案

兩者皆以 localStorage 儲存。

### UI 與語言

介面語言使用全繁體中文。

首頁直接作為行程偏好輸入頁，不做行銷式 landing page。

UI 方向：

* 質感文青
* 字體優美
* 色調低飽和
* 簡單、直覺、快速
* 優先讓使用者一眼比較三方案

Figma MCP 可在設計階段參與，但不得影響核心功能完善度。若實作階段無法密集搭配 Figma MCP，先完成可用介面，後續再用 Figma MCP 重新整理 UI。

所有會呼叫 AI、可能等待較久的操作，都需要 loading 動畫或文字狀態，避免使用者以為畫面卡住。

### Stop 數量規則

AI 不可完全自由決定 stop 數量，必須遵守基礎規則：

* 2-4 小時：2-3 個 stop
* 4-8 小時：3-4 個 stop
* 8-12 小時：4-5 個 stop
* 12-24 小時：5-6 個 stop

依條件微調：

* 不要太累：-1
* 拍照優先：-1
* 美食優先：+1
* 5 人以上：-1

最後 stop 數量必須限制在 2 到 6 之間。

每個方案的 stop 需要有類型，不可只決定數量。至少包含：

* 主活動點
* 餐飲點
* 收尾 / 過渡點

避免產生同質性過高的行程。

### 發布與 PWA

最終目標是可在手機上執行的網頁式 App，並預計使用 PWA 方式讓同組組員能方便使用。

若發布流程不複雜可後續實作；若需要架構預留，開發時應先保留必要插口。

## 2026-04-15 總覽補充（二）

### Gemini API 與前後端邊界

正式 API key 不可直接放在前端程式碼中。

本專案整體仍以前端 MVP 為主，但需要預留未來可切換為 thin proxy / backend 的結構。

AI 呼叫邏輯必須抽成獨立 service / interface，不可寫死在 React 元件內。

若開發初期為了速度暫時由前端直連 Gemini API，可以接受，但程式結構必須能很容易切換成 proxy 模式。

正式分享或發布前，Gemini API 必須改走 backend / proxy。

### AI 回傳格式

AI 回傳格式採用嚴格 JSON 解析。

Gemini prompt 必須要求模型只回傳 JSON，不回傳 markdown、不回傳解釋文字、不包 code block。

前端需對 AI 回傳結果做解析與基本格式檢查。若解析失敗或格式不符，需顯示錯誤狀態，讓使用者可以重新分析。

### 大眾運輸估算

MVP 階段不串接 Google Maps API 或交通 API。

大眾運輸時間先由 Gemini 根據常識與可推估資訊產生，UI 上應視為 AI 預估。

後續若導入 Google Maps API，再將交通時間改為真實資料。

### 交通超時警告

切換交通方式後，若造成整體行程時間過長，需警告使用者。

警告門檻：

* 總時間超過使用者設定時間的 20%
* 或總時間超過使用者設定時間 30 分鐘以上

兩者任一成立即顯示警告。

### PWA 階段

PWA 先列入 Phase 6 優化階段。

Phase 1-5 優先完成核心流程。Phase 6 再處理 mobile polish、manifest、icon、install prompt 等基礎 PWA 設定。

發布流程可後續處理，但專案結構需保持部署友善。

### Figma MCP 時機

早期開發先以功能成型為主。

核心流程穩定後，再使用 Figma MCP 重新整理視覺設計。

若 UI 顯示資訊、頁面布局、卡片資訊層級等規格不明確，每個 phase 開始前都必須先與使用者確認，不可自行大幅假設後直接實作。

## 2026-04-15 Phase 3 開工前確認

### Phase 3 範圍

Phase 3 實作三方案生成，開始接上 Gemini 2.5 Flash、嚴格 JSON schema、結果頁三方案卡片，以及送出後從 loading 進入結果頁。

Phase 3 不做詳情頁完整時間軸、不做收藏、不做雨天切換。這些留到 Phase 4 / Phase 5。

### AI 與錯誤處理

表單送出後呼叫 AI service，使用 `gemini-2.5-flash`。

Gemini prompt 必須要求只回傳 JSON，不可 markdown、不包 code block。

前端需解析 JSON 並做基本格式驗證。若 API 失敗或格式錯誤，顯示錯誤狀態與兩個按鈕：

* 重新分析
* 回到表單

若 `.env` 尚未設定 API key，需顯示明確錯誤。

開發階段可以前端直連 Gemini；正式分享或發布前必須改走 thin proxy / backend。

### 結果頁

成功生成後進入 `/results`。

三張方案卡排序固定為：

1. 保守型
2. 平衡型
3. 探索型

結果暫存於 `sessionStorage`，避免頁面切換後立即遺失。

### 結果卡片資訊

每張卡顯示：

* 方案名稱
* 方案副標題
* 方案風格
* 一句 AI 摘要
* 總時間
* 預估預算
* 預設交通方式
* stop 數量
* 前 2-3 個 stop 名稱
* 選擇此方案按鈕

完整地址、每段交通、完整雨天方案留到 Phase 4 詳情頁。

因每個方案都一定會準備雨天備案，結果卡片不顯示「已準備雨天備案」。雨天備案等使用者選擇方案後，再於詳情頁提供切換。

使用者選擇方案後，詳情頁左上角需提供「重新選擇」按鈕，回到三方案頁重新選擇。

方案標題採「短標題 + 副標題」結構，避免手機版或窄卡片出現奇怪斷行。

AI 回傳規則：

* `title`：4-8 個繁體中文字，不可超過 8 字，不可塞入多個地名。
* `subtitle`：8-18 個繁體中文字，可包含主要地點或亮點。
* `summary`：30 字內的一句描述。

卡片與詳情頁大標題使用 `title`，地點或亮點補充放在 `subtitle`，完整地點資訊留在 stops。

### 交通方式
