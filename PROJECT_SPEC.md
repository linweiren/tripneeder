## 2026-05-05 送出前定位確認 UX 規格

* 首頁按「立即出發」時，若需要使用目前位置，前端只需等到第一個可用 `lat/lng` 座標即可跳出「確認生成行程 / 確認並扣點」對話框；不得等待 `/api/geocode` 反查地名完成後才顯示扣點確認。
* 反查地名屬背景補充流程，可在扣點確認顯示後或生成流程期間完成；生成請求可只帶座標與空地名，後端仍可用座標作候選池與距離約束。
* 背景 geocode 回來時只能補同一組座標的地名；若使用者在等待期間手動改起點或重新定位，舊 geocode 結果不得覆蓋新的 `location`。
* 送出流程仍需保留安全線：若完全無法取得座標，才提示使用者改用指定起點，不應先跳扣點確認。

## 2026-05-05 跨日餐期與雨備去重規格

* 餐期需求仍維持「只要使用者時間窗與餐期有重疊就啟用」；但跨日行程必須保留餐期發生的實際時間軸。例：`14:30-11:30` 的午餐是隔天 `11:00-13:00`，不得被當成當天上午餐期。
* `getRequiredMealWindows` 回傳的 `start` / `end` 需是實際行程時間軸分鐘數，可大於 `24 * 60`。補餐插入、餐飲對齊、候選營業時間檢查都要使用這個實際 occurrence。
* `food` stop 只要與對應餐期有任意重疊即可視為對齊；但若跨日餐期在隔天，必須與隔天餐期重疊，不能拿當天同名餐期誤判。
* 雨天備案不得出現同一個 `placeId` 的重複站點；若沒有 `placeId`，以正規化後的 `name + address` 作為去重 key。
* `/api/generate-trip-details` 僅在雨備驗證後對 `rainBackup` 去重，主行程 `stops` 不得因此被替換或重排。去重後若雨備不足 2 站，維持既有品質規則：清空雨備並讓前端顯示「無符合雨備」。
* 已加入單元測試覆蓋 `14:30-11:30` 跨日餐期：應依序需要晚餐與隔天午餐，且隔天午餐的 target minute 必須是 `24:00 + 12:00`。

# 2. 產品定位與核心目標

**產品定位**：Tripneeder 是「使用者已經在現場 / 附近，快速產生接下來幾小時行程」的工具，不是完整旅遊規劃 AI。

**核心目標**：使用者輸入目前位置與可用時間後，AI 產生三個可比較的一日行程方案，使用者可查看詳情、調整交通、收藏方案。

**產品邊界**：
* 主流程是「即時行程決策」，不包含訂票、付款。
* AI 回傳需符合嚴格 JSON schema。
* 僅支援登入使用者（必須有帳號才能使用核心功能）。

# 3. 目前功能重點 (V1.0 穩定版)

### 3.1 核心頁面 (全頁面極速加載)
* **首頁 (現場快速模式)**：主 CTA 為「使用目前位置」，支援時長 Chip (1h/2h/3h/半天/全天/自訂)。
* **結果頁**：三方案骨架卡片串流顯示。
* **詳情頁**：時間軸、雨天備案、交通切換、拖曳排序、自動補完細節。
* **個人化中心**：收藏、最近生成、人設設定、點數中心（全部採用快取預載，點擊即顯示）。
* **人設設定重設**：使用者可一鍵清除個性化 override；資料庫 persona 欄位回到未設定狀態，`persona_people` 清為 `null`，人數欄顯示未設定，AI 生成時依既有 fallback 使用系統預設 2 人。
* **首頁偏好來源提示**：首頁「更多偏好設定」需顯示目前偏好來源摘要；展開後逐項標示本次設定、個性化或系統預設，讓使用者理解 AI 生成套用的 fallback 狀態。

### 3.2 AI 生成邏輯
* **兩階段生成**：骨架串流 + 詳情補完。
* **地理約束**：起點 2km 約束、Google Places 驗證與 Search-Inject 修正。
* **權重規則**：進階表單 > 個性化人設 > 系統預設。
* **營業時間約束**：Google Places 候選需讀取 `currentOpeningHours` / `regularOpeningHours`；只有具備可解析營業時間的地點可進入候選池與最終方案；每站停留結束需早於打烊前 30 分鐘，未知營業時間視為不可排；最終不符合營業時間的方案不展示給使用者。
* **Places 與營業時間驗證順序**：串流、重試與補案階段需先用 Google Places 解析並正規化地點（取得穩定 `placeId`、名稱與地址），此步驟不做營業時間擋案；接著才用正規化後的 `placeId` 於本地計算動態時間軸並驗證營業時間，避免用未解析或被 Google 替換前的地點資料選錯起點。
* **凌晨低活動時段**：若使用者時間窗從凌晨延伸到早晨（例如 02:00-12:00），02:00-06:00 視為低營業密度緩衝，不強制計入最低行程覆蓋率，也不得當作可塞真實景點的時長上限；實際 stop + transport 總分鐘數需落在 06:00 後有效活動窗的最低覆蓋與最大容量內。候選池可用性、`bestSlots`、Prompt、Places 驗證與 final repair 都必須使用同一個有效活動窗。營業時間驗證不得固定把真實景點排在 06:00；後端需一次取得各站 Google 營業窗，於本地換算「整條行程起點可落入的時間區間」並取交集，選擇第一個符合全路線營業窗與打烊前 30 分鐘規則的開始時間；開始時間以 10 分鐘為本地排程粒度，不得為了試算時間軸重複呼叫 Places API。未知營業時間仍維持硬擋。
* **短行程跨清晨例外**：若短行程從凌晨跨到 06:00 後，但 06:00 後剩餘時間不足以排至少兩個 40 分鐘站點與基本交通（約 98 分鐘），不得把 06:00 前全部視為緩衝，否則會把 3 小時行程壓成 30 分鐘而導致候選池歸零；此時候選池、最低覆蓋率與時間軸搜尋需以完整使用者時間窗計算，只接受真正在凌晨/清晨可營業且可完整停留的候選。
* **實際行程起點同步**：若後端為了營業時間合法性把方案時間軸平移到動態起點，必須在 `TripPlan.scheduleStartTime` 寫入最終驗證採用的實際開始時間；前端詳情頁、最近紀錄與收藏顯示時間軸時需優先使用 `scheduleStartTime`，不得用原始使用者 `startTime` 重畫出未通過營業時間的錯誤時段。
* **串流合法案保護**：若串流過程已成功驗證某個方案 ID，但最終 JSON 解析出的 `plans` 少了該 ID，後端必須把已驗證方案補回 finalResponse 再進 final repair，避免「分析串流看得到合理方案，最終展示消失」。
* **分析完成不可被扣點或收尾流程卡住**：三案 final repair 可並行驗證以避免 Google Places / Routes 串接拖慢整體流程；Supabase 扣點 RPC 必須有 timeout；前端收到 `done` 事件後即可結束分析等待，不得繼續等 NDJSON 連線自然關閉。
* **生成串流中斷提示**：若 `/api/generate-trip` 或詳情補完 fetch 因本地 dev server 重啟、網路中斷或串流連線被切斷而失敗，前端不得直接顯示瀏覽器原生 `Failed to fetch`，需轉成可理解的繁體中文錯誤訊息。
* **生成 JSON 容錯**：骨架生成串流若最後無法解析成完整 JSON，後端需先保留已串流驗證通過的方案；若沒有任何可用 partial plan，需使用同一份 prompt 做一次非串流 structured-output fallback，並記錄 `[trip-json-parse-failed]` 的輸出長度、finish reason 與尾端片段，避免使用者直接遇到「無法解析 AI 回傳的 JSON」且無診斷資訊。
* **三方案補齊策略**：若最終 Routes 與營業時間驗證後少於三案，系統只針對缺少的方案 ID 進行 AI 補齊重產；最終補齊階段需逐一針對缺案 ID 生成，降低模型一次補多案的失敗率；補齊方案仍需通過 Places、Routes、營業時間與硬性節奏品質驗證。
* **補齊 ID 收斂**：AI 補缺案時只能回傳缺少的方案 ID，補案 structured-output schema 也必須只要求缺案數量，不可沿用主生成「固定三案」schema；若模型內容可用但誤把 `id` / `type` 寫成其他主方案 ID，後端需在驗證前收斂為指定缺案 ID 再驗收，避免合法補案因 ID 不符被丟掉。補案 title 不得使用「方案一 / 方案二 / 方案三」等排序字樣。
* **補案修復順序**：AI 補缺案通過 Places 基本解析後，不得因「行程時長不足、短站、可透過時間軸平移修正的 outside opening hours」在前置檢查直接丟棄；這些可修復問題必須交給 final repair 補長、補站、位移與重驗。查無地點、第一站超距、停業、未知營業時間等不可修復問題仍維持硬擋。
* **合法原案保護**：串流或最終解析中曾被標為失效的方案，若後續完整版本已通過硬性驗證，必須從失效清單移除；AI 重試結果只有在通過硬性驗證時才可覆寫既有方案，避免壞補案替換掉原本合法的 AI 方案。
* **最終修復最小化**：final repair 必須先用保守模式驗證 AI 原案，只補齊 Google 地點資訊與交通段，不得主動替換時段候選、移動餐飲站、加站、拉長停留或裁切尾站；保守模式已通過 Places、營業時間、超時、最低覆蓋率與短站驗證時，需直接保留原案。只有保守模式真的不合法時，才可進入覆蓋率修復（時段替換、餐飲位移、加站、拉長、裁切）。
* **最終驗證順序**：delivery 前需先檢查可修復的硬性節奏問題（短站、低於最低覆蓋率、超出可安排時間），必要時先做覆蓋率/容量修復，再用修復後時間軸驗營業時間；避免把過長或短站方案用錯誤時間軸驗成「不在營業時間」。
* **覆蓋率一致性**：Prompt 顯示給 AI 的最低實際行程長度必須與後端硬性驗證使用同一套覆蓋率公式（≤4h: 70%、≤8h: 75%、≤12h: 80%、>12h: 70%），避免模型被要求產出超過後端實際策略的過長行程，或後端誤殺符合 prompt 的方案。
* **少案提示與紀錄**：若營業時間或路線條件導致最終少於三案，僅提示「目前附近營業中的景點較少，僅 X 個可用方案」；最近生成只能保存最終展示的可用方案，不保存被排除的中間方案。由於主方案 ID 固定為 `safe` / `balanced` / `explore`，每次成功生成最近紀錄時需先清掉上一輪三個固定 ID 的 recent records，再寫入本輪最終可展示方案，避免上一輪未展示的第三案殘留。
* **候選池品質優先**：Google Places 候選需先經本地角色分類、時段標註與評分排序，再分區注入給 AI（第一站、餐飲、主活動），讓 AI 從乾淨候選中組合而非自由發明。
* **候選池為零不生成**：若 Google Places 候選池在目前時間窗下為 0，或有座標時起點 2 公里內沒有可驗證第一站候選，後端不得呼叫 AI 自由生成，也不得讓 AI 用泛稱硬湊；需直接回報候選不足並建議調整時間、起點或行程長度。
* **餐飲候選穩定性**：餐飲候選需細分 `cafe`、`dessert`、`restaurant`、`snack`；使用者未選美食優先時，短行程的 food stop 應偏好可坐下休息的咖啡、甜點或餐廳，降低傳統小吃、夜市、市場攤位權重；候選排序需用分數、距離、評分、名稱/placeId 做穩定 tie-breaker。
* **餐飲時段約束**：未勾選「不吃正餐」且行程與 11:00-13:00 或 17:00-19:00 有任何重疊時，就視為涵蓋對應餐期；正式餐廳類 food stop 只需與對應餐期有重疊即可，不要求命中餐期中心或完整覆蓋餐期；但仍不得放在清晨、完全不重疊餐期，或餐廳尚未營業的時段。候選 `bestSlots` 需以整個時段區間判斷，不可只用單一 slot 起點；若 AI 選到合法餐廳但完全落在餐期外，final repair 可優先保留同一餐廳並把 food stop 往餐期/尾段移動後重算交通與營業時間。
* **餐飲候選判斷**：補餐、移餐與候選排序也必須沿用「只要與餐期重疊」的語意；不得要求餐飲停留完整落在 11:00-13:00 或 17:00-19:00 內，避免可用餐廳被過度保守的候選篩選排除。
* **雙餐期覆蓋**：未勾選「不吃正餐」且行程時間窗與午餐、晚餐兩個餐期都有任何重疊時，最終方案至少需有兩個餐飲停留，並分別與午餐、晚餐時段有重疊；若 AI 原案缺餐或兩個 food stop 都落在同一餐期，這屬可修復硬性問題，需進入 coverage repair 補餐、移餐、重算交通與重驗營業時間。補長、候選替換或交通重算會改變抵達時間，因此 coverage repair 後段需再次移餐並再次驗證候選營業時間；`food` stop 不可被時段候選替換成非餐飲地點。餐飲只要與餐期重疊就視為通過餐期覆蓋，不得因重疊時間長短、不是餐期中心或「看起來太早/太晚」而硬殺方案；但補餐/移餐的排序目標可放在餐期中段，避免本地估算與 Routes 重算後整段餐飲落到餐期外。
* **候選時段對齊**：final repair 補齊 Google 候選資訊時，不得只因地點存在就保留；候選能否保留以「預估抵達時間 + 停留時間可完整落在 Google 營業窗內」為準，`bestSlots` 只作為粗略排序與補案提示，不可因 early / middle / late 邊界誤殺精確時間合法的地點；若精確時間無法完整停留，需優先替換成同類型且該時段可完整停留的候選，避免把 06:00 才開的地點排在 04:50。
* **補長後再次時段對齊**：final repair 在補站、拉長停留、容量裁切後，需用修復後的實際 `stop.duration` 與 `transportSegments.duration` 再跑一次候選時段對齊並重算交通；若替換候選導致交通時間改變，需用重算後時間軸再檢查少量迭代，避免原本白天合法的早關門景點在長時段補長後被推到傍晚，最後才因營業時間硬擋導致整案被排除。
* **餐飲不硬塞**：骨架生成不得要求每個方案都強制涵蓋 `food` 與 `ending_or_transition`；每案至少需要 `main_activity`，餐飲只在行程涵蓋餐期且候選營業窗可支撐至少 45 分鐘停留時安排。若餐期只擦到使用者時間窗尾端或餐飲候選不足，應優先產出合法主活動方案，不可為了湊類型導致三案被營業時間或最低覆蓋率擋掉。
* **跨方案多樣性**：跨方案地點重複屬軟性規則；最終展示前不得為了避開前案 `placeId` 主動替換已合法的 AI 原始方案，也不得因局部或高度重複直接刪除已通過硬性驗證的方案。後端僅輸出重複診斷 log；若方案因長時段不足進入 coverage repair，可依 `safe` / `balanced` / `explore` 對補長候選與補餐候選做輕量輪替，讓補出來的長行程不要每案都拿同一批候選前段，但找不到替代時必須回退合法候選，不得為了多樣性硬殺方案。
* **本地補案組合器**：本地補案預設停用，僅在 `ENABLE_LOCAL_FALLBACK_REFILL=true` 時作為 opt-in fallback。若啟用，後端可用 Google 候選池產生多組 2-3 站備選路線並依序驗證；本地補案仍需通過 Places、Routes、營業時間、短站、超時與最低覆蓋率等硬性驗證。補案 `id` / `type` 必須等於缺少的方案 ID（`safe` / `balanced` / `explore`），標題不得使用容易與排序混淆的「方案一/二/三」字樣。
* **候選池補洞**：當行程涵蓋晚上或深夜時，Google 搜尋需額外補夜間景點、夜市、晚間咖啡、24 小時營業等查詢，以提高晚間可用方案數。
* **缺口式擴容**：候選池不可用單純重複擴半徑的方式暴力增加 API；第一輪以 5km 核心查詢建立候選池，若本地分類後缺第一站、餐飲、主活動、室內或夜間候選，才使用 10km 與缺口型 query 補查，補查 query 有上限並以 `placeId` 去重後重新評分。
* **驗證可觀測性**：營業時間排除方案時，後端需記錄方案 ID、站點、推算抵達/離開時間與 Google 營業窗，方便調整候選與驗證規則。
* **時間上限收斂**：最終方案不得超過使用者指定可用時間；若 Routes 後超時或尾站停留過短，系統可裁掉最後一站並重算 Routes，只要裁剪後仍符合最低覆蓋率；短行程應優先拉長有效停留時間，目標貼近使用者可用時間約 90%，避免明明有空檔卻留下 20 分鐘短站。
* **長時段多站策略**：8 小時以上不應固定 3 站或用少數站點硬拉長；4-8h 通常 3-5 站，8-12h 通常 4-6 站，12-16h 通常 5-7 站，16h 以上至少約 6 站。final repair 補足長度時需優先加入非餐飲主活動/購物/開放空間候選，再合理拉長停留；不得把單一景點拉到明顯超過場域特性的長度來滿足最低覆蓋率。
* **行程節奏品質**：方案需檢查停留時間是否合理、是否連續安排過多公園/戶外開放空間、是否用過多短站湊路線，以及交通時間占比是否過高；非餐飲景點原則上至少 40 分鐘，餐飲至少 45 分鐘，不得因標成 `ending_or_transition` 就繞過短站檢查。Places 驗證、營業時間、超時、低於最低覆蓋率與短站屬硬性擋案；交通占比與同質景點過多只做 log / 診斷 / 補案參考，不得觸發最終刪案。
* **中段主景點節奏**：中段 `main_activity` 若低於 45 分鐘可列為 soft diagnostic，但只要仍符合非餐飲硬性最低 40 分鐘，不得作為最終硬刪方案理由。
* **方案品質觀測**：後端需輸出 plan quality summary，包含方案階段、總分鐘數、可用時間、站點數、交通占比、每站角色與節奏問題，方便調整候選池與驗證門檻。
* **少案診斷觀測**：當最終少於三案時，後端需輸出缺少方案 ID、補案失敗原因、已展示方案 ID 與候選池角色數量，讓後續可判斷是候選不足、模型未回傳、營業時間、路線或節奏品質造成。
* **零案錯誤可診斷**：若最終 0 案，後端不可只回傳通用錯誤；需在 console 輸出 `[no-available-plans]` 結構化原因，並把各方案最終排除原因帶回前端錯誤訊息，方便直接定位是營業時間、第一站距離、時長、Routes 或候選不足。
* **雨天備案品質**：詳情補完的雨天備案需通過 Places、營業時間、最低站點數與最低時長檢查；若補完結果只剩單一短站或低於行程最低覆蓋率，後端不可展示該雨天備案。
* **雨備可用狀態**：詳情頁只有在雨天備案具備至少 2 個有效站點時才可切換；若後端清空不合格雨備，按鈕需反灰不可點並顯示「無符合雨備」。
* **換一站交通策略**：詳情頁替換單一景點時，不應立即呼叫 Google Routes 或扣點；替換後只將相鄰交通段標為本地估算並更新總時長，使用者若要精準路線需再按「重算交通」。

### 3.3 數據同步與效能
* **全域預載 (Pre-fetch)**：使用者登入成功後，立即異步抓取所有用戶資料（點數、人設、紀錄）。
* **本地優先 (Optimistic UI)**：任何操作（如收藏、儲存人設）先反映在 UI 並更新本地快取，背景才與伺服器同步。

## 2026-05-05 生成結果與歷史紀錄隔離規格

* 生成結果頁與最近生成/收藏詳情頁不得只用固定方案 ID（`safe` / `balanced` / `explore`）共享同一份詳情狀態。詳情補完與 loading/error/complete 狀態必須依來源 scoped：目前生成使用 `generated:<planId>`，歷史紀錄使用 `<source>:<recordId>`。
* 使用者在新一輪生成期間瀏覽最近生成或收藏時，歷史紀錄寫入的 `detailPlan` / `detailInput` 不得被目前生成流程讀取。生成結果的詳情補完只能使用 `generatedPlans` 與 `lastInput`；最近生成/收藏只能使用該紀錄自己的 detail/input。
* 歷史紀錄詳情補完或換站/重算等更新，只能回寫該筆 recordId；不得因 `balanced` 等固定 ID 相同而覆寫目前生成中的方案或其他歷史紀錄。
* 詳情頁「換一站」候選池必須使用當前頁面來源的 input，不得 fallback 到上一個瀏覽過的 detail input，避免 3 小時行程套到 13 小時行程的候選與時間條件。

## 2026-05-05 凌晨短行程候選池規格

* Google Places 候選池不得只因營業時間與使用者時間窗有邊界接觸就視為可用；候選必須能在使用者時間窗內完整停留至少 40 分鐘，餐飲候選至少 45 分鐘，且離開時間需早於打烊前 30 分鐘。
* 若 3 小時等短行程完全落在 06:00 前低營業密度時段，AI 只能使用已知在該時段營業且可完整停留的候選；06:00 才開門、但使用者時間窗 06:00 結束的地點不得注入給 AI，也不得安排到最終方案。
* 若 3 小時短行程只跨過 06:00 一小段，06:00 後不足以排完整路線時，不得將排程起點固定平移到 06:00；應搜尋凌晨/清晨與 24 小時候選，若仍無可驗證候選則直接回報候選不足，不得自創「湖畔咖啡館」「在地小吃店」等泛稱。
* 短行程若候選同時包含 24 小時與 06:00 後營業地點，後端需依站點預估抵達/離開時間套用精確營業時間；`bestSlots` 僅作粗略排序提示，不可覆蓋精確時間判斷。早段/中段不可保留精確時間無法完整停留的地點；若可替換候選存在，final repair 必須先替換再驗營業時間。

## 2026-05-05 長時段補長與餐飲時間規格

* Final repair 若因行程時長不足進入 coverage repair，補站目標不得只壓在最低覆蓋線，需以後端 target coverage ratio 作為補長目標並受使用者可用時間上限約束，避免 10 小時行程補完後仍只差數分鐘而被硬擋。
* 長時段補站需優先使用非餐飲、非短停留候選；若候選池過窄，不得因 short-visit 分類完全放棄補站，可退而使用未用過且營業時間合法的候選，以「合理補足」優先於 0 案。
* 餐飲站位移不可只用固定百分比；當行程涵蓋午餐或晚餐時，需依有效時間軸推估餐飲抵達時間，將 food stop 插入最接近 11:00 或 17:00 的位置，再重算交通與營業時間。
* 若 coverage repair 仍失敗，前端與 log 應回報 coverage attempt 的實際失敗原因，而不是只回報第一次保守驗證的舊錯誤。
* 後端用 Google 候選補正站點時，必須同步校正 stop 類型；若命中餐廳、咖啡、烘焙等 food candidate，即使 AI 原本標成 `main_activity` 或 `ending_or_transition`，也要改成 `food` 並套用餐飲最短停留與餐期位移規則。
* 長時段 coverage repair 的補站候選池需合併 `otherCandidates`、`firstStopCandidates` 與 `allCandidates` 並以 `placeId` 去重，避免單一分區太窄導致明明有合法候選卻補站失敗。
* Food candidate 判斷需優先使用後端候選池已推論出的 `role=food` / `foodSubtype`，不得只依賴 Google `types`；餐酒館、咖啡、早午餐等即使 Google types 不完整，也要套用餐飲規則。
* 長時段行程涵蓋午餐或晚餐且使用者未勾選不吃正餐時，若 AI 原案沒有任何 food stop，coverage repair 可從合法 food candidates 補入一站並排到餐期附近；若沒有合法 food candidate，才保留無餐飲方案，不得因此硬殺。
* AI 補缺案若回傳內容合法但 `id` / `type` 不等於指定缺案 ID，後端需先改成指定 ID 再進 Places、營業時間、Routes 與硬性節奏驗證；title 若含方案序號需改成不含序號的安全名稱。補案 schema 必須允許只回傳 1 個缺案，避免模型為了滿足三案 schema 產生錯 ID 或重複方案。
* AI 補缺案若前置驗證只剩可修復節奏問題（短站、低於最低覆蓋率、超出可安排時間）或可透過整體時間軸位移修正的 outside opening hours，不可直接判定失敗；需進入 final repair 後再依修復結果決定是否展示。

## 2026-05-05 生成流程重構與測試規格

* 時間窗、跨日、凌晨有效活動窗、覆蓋率、餐期重疊、固定方案 ID 等純規則需集中於 `api/_lib/trip-planning-rules.ts`，不得分散在 `/api/generate-trip` 主流程內重複實作。
* 方案總時長、預估交通、基本停留時間、長行程最低站數等純度量需集中於 `api/_lib/trip-plan-metrics.ts`。
* 站點抵達時間、可用性 early / middle / late slot、餐期插入位置與整案時間軸推算需集中於 `api/_lib/trip-timeline.ts`。
* 短站、必要餐期覆蓋、交通占比、同質景點與站點節奏角色等品質規則需集中於 `api/_lib/trip-quality-rules.ts`；其中交通占比與同質景點只屬 soft diagnostic，不得作為最終硬刪方案依據。
* 方案顯示順序、補長目標、跨方案候選輪替與 rotated pick 策略需集中於 `api/_lib/trip-repair-strategy.ts`，避免主流程再次散落 safe / balanced / explore 對應與補案選擇邏輯。
* `/api/generate-trip` 可負責 orchestration、串流、AI、Places、Routes、Supabase 與修復流程，但純規則需優先抽到可單元測試的 `_lib` 模組。
* 新增或調整硬性生成規則時，需優先補 `npm run test:unit` 的本地測試；除非測試範圍真的涉及 Google Places / OpenAI 回應品質，否則不得用真實 API 測試取代單元測試。
* 單元測試目前覆蓋：固定三方案 ID、跨日時間窗、凌晨有效活動窗、覆蓋率門檻、補長目標、餐期需求任意重疊啟用門檻、餐飲任意重疊餐期、方案實際時長、預估交通、長行程最低站數、基本停留門檻、景點類型停留上限、時間軸抵達推算、餐期插入位置、品質 soft diagnostic、雙餐期必要餐飲與方案順序/輪替策略。

## 2026-05-05 詳情補完效能與資料邊界規格

* 詳情補完只負責補主方案 stop 描述、交通 label 與可用雨天備案，不得重寫已驗證的主方案 stop identity、placeId、地址、停留時間、順序、totalTime 或主交通 duration。
* `/api/generate-trip-details` 不再對主行程重跑 Places 驗證；主行程合法性以 `/api/generate-trip` 最終驗證結果為準。詳情補完只驗證 rainBackup，避免打開詳情頁時破壞已合法方案。
* 詳情補完不重算主行程 Google Routes；只有主交通段缺漏或 from/to 對不上時才補算。雨備若存在且至少 2 站，才補算雨備交通。
* 詳情補完 AI 回傳採增量 schema：`stops` 只回 `id + description`，`transportSegments` 只回 `fromStopId + toStopId + label`，主方案完整資料由後端用原骨架合併保留。
* 可用時間超過 8 小時的長行程，詳情補完預設不生成雨天備案，回傳空 `rainBackup`，前端維持「無符合雨備」不可點狀態；避免產生明知無法達到長行程覆蓋率的雨備後又被刪除。
* 詳情補完需輸出 `[detail-performance]` 階段 log，至少包含 location、persona、indoor candidates、OpenAI、parse、rain validation、routes、done 的耗時，便於後續追查慢點。
* 單元測試需覆蓋詳情 parser 的增量合併行為，確認 AI 只補描述/label 時不會覆蓋已驗證主方案資料。
